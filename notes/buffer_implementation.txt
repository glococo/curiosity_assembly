1. HAL_DOUBLEBUFFER.S (Ping-Pong Architecture)
  This implementation uses two identical memory blocks (Active and Spare) and a descriptor to manage them. It is designed for high-throughput,
  block-oriented data handling.

   * Speed (Strongest):
       * Interrupt Latency: HAL_DOUBLEBUFFER_PUSH is extremely fast. It performs a direct store with post-increment (st X+) and a simple counter
         comparison.
       * Zero-Copy Potential: While the current WRITE_BYTE function pulls bytes one by one, the architecture allows the "Spare" buffer to be handed
         off entirely to a peripheral (like a DMA or a high-speed loop) while the "Active" buffer continues to collect data.
   * Memory Usage (Weakest):
       * High Overhead: Requires double the SRAM for data (2x size) plus a 9-byte descriptor.
   * Characteristics:
       * Block-Oriented: It only signals "Ready" (r18=1) once a full block is captured. This is ideal for processing data in chunks (e.g., FFT,
         display frames, or block-based communication protocols).
       * Risk of Overrun: If the consumer does not finish draining the Spare buffer before the Active buffer fills up and triggers a FLIP, data in
         the Spare buffer will be overwritten.

  2. HAL_DEVICEBUFFER.S (Circular/Ring Buffer)
  This is a classic circular buffer implementation using a power-of-two size to optimize wrap-around logic.

   * Memory Usage (Strongest):
       * Efficiency: Requires only 1x size for data plus a 5-byte descriptor.
   * Features & Flexibility (Strongest):
       * Stream-Oriented: Data can be Pushed and Pulled byte-by-byte asynchronously. There is no "waiting for a block to be full" requirement.
       * Status Methods: Includes helper functions like IS_EMPTY, IS_FULL, and IN_USE, making it easier to manage flow control.
   * Speed (Moderate):
       * Masking Logic: Every PUSH and PULL requires loading the Mask, Head, and Tail, performing an increment, and a bitwise AND. While optimized,
         it involves more instructions than the double-buffer push.
   * Constraints:
       * Power-of-Two Size: The buffer size must be a power of two (64, 128, etc.) because the wrap-around logic relies on bitwise masking instead
         of conditional branching or division.

  Summary Comparison Table


  ┌─────────────────┬───────────────────────────────────────┬──────────────────────────────┐
  │ Feature         │ HAL_DOUBLEBUFFER                      │ HAL_DEVICEBUFFER             │
  ├─────────────────┼───────────────────────────────────────┼──────────────────────────────┤
  │ Primary Use     │ High-speed block transfers / DMA-like │ General purpose byte streams │
  │ ISR Performance │ Fastest (Simple counter + st X+)      │ Fast (Incremental + Masking) │
  │ SRAM Cost       │ High (2x size + 9 bytes)              │ Low (1x size + 5 bytes)      │
  │ Data Access     │ Block-based (Full/Empty)              │ Byte-based (Continuous)      │
  │ Complexity      │ Higher (Requires FLIP management)     │ Lower (Standard Head/Tail)   │
  │ Size Constraint │ 2 to 256 bytes                        │ Power-of-Two only            │
  └─────────────────┴───────────────────────────────────────┴──────────────────────────────┘

  Recommendation:
   * Use HAL_DOUBLEBUFFER for high-speed peripherals (like USART at high baud rates or SPI) where you want to minimize the time spent in the ISR and
     process data in fixed-size packets.
   * Use HAL_DEVICEBUFFER for general-purpose UIs, command parsing, or low-to-medium speed communication where memory efficiency and asynchronous
     byte access are more important.
